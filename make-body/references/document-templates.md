human-confirmed. `{Term}` is allowed only in the candidate queue.


Rules:

- `Subject` is plain text, has no `[]` or `{}`, and becomes `## [Subject]` on Yes.
- Content contains one independently confirmable claim or one Given/When/Then flow.
- Evidence points only to in-scope code artifacts.
- A ready candidate has an empty `Blocked by` section or a bare `-`.
- `extract-typescript-candidates.py` may create only code-literal System scaffolds in this file;
  review their cited source before approval or auto-approval.

## MAKE_BODY_CODE_FACTS.md

Generated by `scripts/collect-typescript-facts.py`. This is a regenerable source index, not
canonical domain language. Every fact retains its source pointer and lexical kind; inspect the
cited code before using a fact as candidate evidence.

## MAKE_BODY_COVERAGE_IGNORE.md

```md
# MAKE_BODY_COVERAGE_IGNORE

- `src/helper.ts` — utility has no supported lexical entry point
```

Each entry must name an existing in-scope supported source path without facts and give a concrete
reason. Paths are unique. This documents a coverage exception; it does not establish a domain
claim.

## MAKE_BODY_REJECTED.md

```md
# MAKE_BODY_REJECTED

## MR-001

Type:
system-domain

Subject:
Legacy Sessions Table

Reason:
- (free text — why this is not canonical; if it is superseded or reframed, name what replaces it
  inline)

Evidence:
- `path:line` — code the rejection refers to

Replacement:
-
```

`MAKE_BODY_REJECTED.md` is an optional archive. A human appends an `## MR-{nnn}` record only when
they **explicitly reject** a candidate (legacy, accidental, or wrong implementation; an Essential
hypothesis that is not the business meaning; or an ambiguous System candidate ruled unfit). A
candidate merely awaiting human review is **not** rejected — it stays in `MAKE_BODY_CANDIDATES.md`.
`Reason` is free text; do not enumerate fixed subtypes. `Replacement` is optional. `MR-{nnn}`
numbering is monotonically increasing and a removed record does not free its number. Do not move an
`MR-{nnn}` record back to the candidate queue.

`Type`, `Subject`, and each `Evidence` pointer are the rejection key, applied **per pointer**: the
candidate extractors read this archive and never re-seed a scaffold whose `(Type, Subject, Evidence
pointer)` matches a rejected record, and `check-rejected.py` flags any queued candidate that re-cites
*any one* rejected `(Type, Subject, pointer)` triple — even alongside new evidence. A genuinely new
claim is one that shares *no* rejected pointer for that `Type`+`Subject`; rejection binds to each
pointer, not to the full `Evidence` set and not to the name.
